Architecture 15 Nov 2024

Why Clean Architecture Matters More Than Your Framework Choice

Every few months, a new framework promises to solve all your problems. Developers argue ‚Äî React or Vue? Express or Fastify? But here's what I've learned: the framework is the least important decision you'll make. What truly matters is how you organise your code around it.

Clean architecture is about separating concerns clearly. Your business logic should not know or care whether it's being served by an Express route or a serverless function. When logic bleeds into controllers, when database calls sit inside UI components ‚Äî you've created a system that's hard to test, hard to change, and hard to hand off.

The simplest principle I follow: if I can't move a piece of logic to a different delivery mechanism without rewriting it, I've coupled it too tightly. Good architecture makes the plumbing invisible. The business rules are the star ‚Äî everything else is infrastructure.

Start small. Separate your route handlers from your service logic. Keep your database queries in one place. Name things by what they do, not how they do it. You don't need hexagonal architecture on day one ‚Äî you just need to start drawing clearer lines.

The framework will get replaced eventually. Your business logic doesn't have to be rewritten when it does. That's the whole point.

REST API 20 Oct 2024

Designing REST APIs That Don't Break When Requirements Change

Requirements change. That's not a failure of planning ‚Äî it's just how software works. The problem isn't that your API needs to evolve; it's that most APIs are designed with no room to evolve at all. One new field and everything breaks.

The first thing I do is version from day one. Even if you never need v2, having /api/v1/ in your routes costs nothing and buys you everything. It signals to consumers that you take backward compatibility seriously, and gives you an escape hatch when breaking changes are unavoidable.

Resource naming matters more than people think. Keep resources as nouns, not verbs. Use /orders, not /getOrders. Use HTTP methods to express intent ‚Äî GET to fetch, POST to create, PATCH to partially update, DELETE to remove. When your routes read like plain English, they're easier to predict and harder to misuse.

Always return consistent response shapes. A success response should always look like a success response. An error should always look like an error ‚Äî with a predictable structure and a useful message. Inconsistency forces every client to write defensive code for every endpoint.

Design for your consumers, not for your database. The shape of your API response and the shape of your database table are not the same thing. Expose what the client needs, nothing more. Overfetching is a sign that the API was designed from the inside out.

Frontend Performance: The Basics That Actually Move the Needle

Most performance guides jump straight to service workers, edge caching, and tree-shaking. Those things matter ‚Äî eventually. But for the majority of web apps, the biggest gains come from doing the basics correctly. Here's where I start.

Images are almost always the biggest problem. Serve them in modern formats like WebP, size them appropriately for the viewport, and always use loading="lazy" on anything below the fold. A single unoptimised hero image can cost you more than every other optimisation you'll ever make.

Render-blocking resources are the next culprit. Put CSS in the <head>, JavaScript at the end of <body> or marked as defer. The browser can't render what it can't parse, and it can't parse while it's waiting for a script to download.

Reduce unnecessary DOM operations. Each time you read layout properties and then write to the DOM, you risk a forced reflow. Batch your reads, batch your writes. Use DocumentFragment for bulk insertions. The browser is fast ‚Äî don't make it redo work it already did.

Measure before you optimise. Open Chrome DevTools, run a Lighthouse audit, and fix the three highest-impact issues it surfaces. You'll be surprised how far that takes you before you ever need to touch a webpack config.

Git 12 Aug 2024

Git Habits That Make You a Better Collaborator

Git is not just a backup system. It's a communication tool ‚Äî a log of decisions, not just changes. When I look at a commit history and see "fix", "wip", "asdf", I know the developer was using Git as a save button. That's a missed opportunity.

Write commit messages that explain why, not just what. The diff already shows what changed. The message should tell the reader why you changed it ‚Äî what problem it solved, what behaviour it introduced. A good message is one you'll thank yourself for six months later at 11pm.

Commit small and commit often. One logical change per commit. If your commit message needs the word "and", it probably contains two commits. Smaller commits are easier to review, easier to revert, and make git bisect actually useful when hunting bugs.

Use branches for everything ‚Äî even solo work. A branch gives you a safe place to experiment. It keeps your main branch deployable at all times. And when you're ready to merge, you have a clean, reviewable unit of work instead of a wall of unrelated changes.

Learn the commands beyond add, commit, and push. git stash, git rebase -i, git log --oneline ‚Äî these are the tools that separate someone who uses Git from someone who understands it.

Database 01 Jul 2024

Supabase vs MongoDB: Choosing the Right Database for Your Project

The "best database" debate usually misses the point. There is no best ‚Äî there's only the right tool for your specific situation. I've used both Supabase and MongoDB in production, and the choice has never been about which is superior. It's been about which fits the project's shape.

Supabase is a joy when your data is relational. Users have orders. Orders have line items. That structure is well-defined, consistent, and benefits from joins and foreign key constraints. Supabase gives you Postgres, real-time subscriptions, built-in auth, and row-level security ‚Äî all with minimal setup. For structured data and rapid prototyping, it's hard to beat.

MongoDB shines when your data is flexible or hierarchical. Content that varies by type, logs with unpredictable fields, documents that are naturally nested ‚Äî these fit MongoDB's schema-less model well. You're not fighting the database to store what you actually have.

The mistake I see most often: using MongoDB because it feels simpler upfront, then spending weeks working around the lack of relational integrity. Or using a relational database for something genuinely document-shaped and forcing everything into flat tables.

My rule of thumb: if you can draw an entity-relationship diagram that makes sense, use Supabase. If your data resists that diagram, consider MongoDB. Start from the data, not the database.
